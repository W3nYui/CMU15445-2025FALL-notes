### 概述 (Overview)

在此项目中，你将在数据库系统中实现一个 **B+树 (B+Tree)** 索引。B+树是一种平衡查找树，其中内部页面 (Internal Pages) 引导搜索，叶子页面 (Leaf Pages) 包含实际的数据项。索引提供了快速的数据检索，而无需搜索数据库表中的每一行，从而实现了快速的随机查找和高效的有序记录扫描。

你的实现需要支持**线程安全**的搜索、插入、删除（包括节点的拆分和合并），以及一个用于支持有序叶节点扫描的迭代器。你需要完成以下任务：

- **Task #4 - Concurrency Control (并发控制)**

你在 Project #2 中的工作依赖于你在 Project #1 中实现的缓冲池 (Buffer Pool) 和页面守卫 (Page Guards)。如果你的 Project #1 实现不正确，你**必须**先修复它才能成功完成本项目。本项目必须独立完成（即不能组队）。

### 任务 #1 - B+树页面 (Task #1 - B+Tree Pages)

你必须实现以下三个 Page 类来存储 B+ 树的数据。

1. **B+Tree Page (B+树基础页面)**
2. **B+Tree Internal Page (B+树内部页面)**
3. **B+Tree Leaf Page (B+树叶子页面)**

#### 基础页面 (Base Page)

这是一个基类，内部页面和叶子页面都继承自它，并且只包含子类共享的信息。B+Tree Page 具有以下字段：

| 变量名称     | 大小 (字节) | 描述                                      |
| ------------ | ----------- | ----------------------------------------- |
| `page_type_` | 4           | 页面类型 (无效页面 / 叶子页面 / 内部页面) |
| `size_`      | 4           | 页面中当前键值对的数量                    |
| `max_size_`  | 4           | 页面中最大的键值对容量                    |

你必须且只能通过修改其头文件 (`src/include/storage/page/b_plus_tree_page.h`) 和对应的源文件 (`src/storage/page/b_plus_tree_page.cpp`) 来实现 B+Tree Page。

#### 内部页面 (Internal Page)

内部页面（即内节点）存储 $m$ 个有序键和 $m + 1$ 个指向其他 B+ 树页面的子页面指针 (`page_id`)。这些键和指针在内部表示为键/page_id对（`std::pair<KeyType, ValueType>`）的数组。由于子指针的数量比键的数量多一个，因此 `key_array_`（见 `src/include/storage/page/b_plus_tree_internal_page.h`）中的**第一个键被设置为无效键**，并且查找操作应始终从第二个键开始。

在任何时候，每个内部页面都应该**至少半满**。在删除期间，两个半满的页面可以合并，或者可以重新分配键和指针以避免合并。在插入期间，一个已满的页面可以分裂为两个，或者可以重新分配键和指针以避免分裂。这些是你在实现 B+ 树时需要做出的许多设计选择的示例。

你必须且只能通过修改其头文件 (`src/include/storage/page/b_plus_tree_internal_page.h`) 和对应的源文件 (`src/storage/page/b_plus_tree_internal_page.cpp`) 来实现内部页面。

#### 叶子页面 (Leaf Page)

叶子页面存储 $m$ 个有序键和它们对应的 $m$ 个值。在你的实现中，值应该始终是实际元组存储位置的 64 位记录 ID (Record ID) —— 参见 `src/include/common/rid.h` 中的 `RID` 类。叶子页面对键/值对的数量具有与内部页面相同的限制（半满限制），并且在合并、拆分和重新分配键时应遵循相同的操作。

**💡 2025秋季学期重大变化 (延伸实现)：** 在这个项目中，我们将扩展我们的叶子页面实现，增加一个用于**记录近期删除的墓碑缓冲区 (tombstone buffer)**。该墓碑缓冲区存储被删除的键/值数组项的最后 $k$ 个索引。

因此，当从索引中删除一个键时（如果 $k > 0$），它在对应叶子页面中的物理条目**不会被实际删除**，而是将其物理数组索引追加到墓碑缓冲区中。**只有当该叶子页面的墓碑缓冲区内积攒了** $k$ **个条目时，最老的缓冲删除才会被真正应用（物理删除并移动）到键/值数组中。** 这是课堂上讨论的 $B^\epsilon$-tree 的简化版。

- 你必须实现 `GetTombstones()` 来报告给定页面中墓碑所对应的键。
- 但是，`KeyAt` 必须返回给定索引处的物理条目，**无论该条目是否存在墓碑**。

你必须且只能通过修改其头文件 (`src/include/storage/page/b_plus_tree_leaf_page.h`) 和对应的源文件 (`src/storage/page/b_plus_tree_leaf_page.cpp`) 来实现叶子页面。

> ℹ️ **注意：** 尽管叶子页面和内部页面包含相同的键类型，但它们可能具有不同的值类型。因此，它们的 `max_size` 可能会有所不同。 每个 B+ 树叶子/内部页面都对应于从缓冲池获取的物理内存页面（即 `data_` 部分）。每次读取或写入叶子/内部页面时，必须先使用其唯一的 `page_id` 从缓冲池获取该页面，使用 `reinterpret_cast` 将其转换为叶子或内部页面，并在完成读写后 unpin 该页面。

### 任务 #2 - B+树操作（插入、删除和单点查找）

在这个任务中，你的 B+ 树索引需要支持单值的插入 (`Insert()`)、删除 (`Remove()`) 和查找 (`GetValue()`)。

- 该索引**仅支持唯一键 (unique keys)**。如果你尝试将已存在的键重新插入到索引中，不应执行插入操作，且应返回 `false`。
- 你必须且只能通过修改源文件 `src/storage/index/b_plus_tree.cpp` 以及可选的对应头文件 `src/include/storage/index/b_plus_tree.h` 来实现此任务。

如果插入会违反 B+ 树的性质，则应分裂 B+ 树页面（或重新分配键）。此外，**在任何合并、拆分和重新分配操作中，必须维护叶子页面墓碑（及其顺序）**。当一个叶子页面合并（coalesced）或重新分配（redistributed）到另一个叶子页面时，我们将该节点中所有未处理的删除视为比接收方叶子页面中任何未处理的删除**更新**（换句话说：被移入新条目的目标节点，其墓碑应当先被处理）。

如果插入更改了根节点的 Page ID，你必须更新 B+ 树索引的头部页面 (header page) 中的 `root_page_id`。你可以通过访问在构造函数中提供给你的 `header_page_id_` 页面来完成。然后，通过使用 `reinterpret_cast`，你可以将此页面解释为 `BPlusTreeHeaderPage`（来自 `src/include/storage/page/b_plus_tree_header_page.h`）并从中更新根页面 ID。你还必须实现 `GetRootPageId`。

同样地，如果删除键后为了维护 B+ 树的性质需要，你的 B+ 树索引必须支持在页面之间合并或重新分配键。与插入一样，如果根节点发生变化，你必须正确更新 B+ 树的根页面 ID。

我们**强烈推荐**你使用 Project #1 中的 **页面守卫 (Page Guard) 类** 来避免同步问题。你应该根据需要相应地使用 `ReadPage` 或 `WritePage`。

你可以选择使用 `Context` 类（定义在 `src/include/storage/index/b_plus_tree.h` 中）来追踪你读取或写入的页面（通过 `read_set_` 和 `write_set_` 字段），或存储递归传递到其他函数所需的其他元数据。

#### 💡 如果你使用 `Context` 类，以下是一些提示：

1. 你可能只需要在插入或删除时使用 `write_set_`。视你的实现而定，你可能不需要使用 `read_set_`。
2. 你可能希望在 context 中存储根页面 ID，并在修改 B+ 树时获取 header 页面的写锁守卫（write guard）。
3. 要寻找当前节点的父节点，可以查看 `write_set_` 的末尾（back）。它应该包含沿访问路径的所有节点。
4. 你可以使用 `BUSTUB_ASSERT` 来帮助你在实现中发现不一致的数据。例如，如果你想分裂一个节点（根节点除外），你应该确保 `write_set_` 中至少还有一个节点。如果需要分裂根节点，你应该检查 `header_page_` 是否为 `std::nullopt`。
5. 要释放头部页面的锁，只需将 `header_page_` 设为 `std::nullopt`。
6. 要释放其他页面，从 `write_set_` 中 pop 并 drop。

B+ 树对任意键、值和键比较器类型进行了参数化。我们定义了一个宏 `INDEX_TEMPLATE_ARGUMENTS`，它为你生成模板参数声明：

```
template <typename KeyType, typename ValueType, typename KeyComparator>
```

类型参数为：

- `KeyType`：索引中每个键的类型。在实践中，这将是一个 `GenericKey`。`GenericKey` 的实际大小会有所变化，并且通过其自身的模板参数指定，该参数取决于被索引属性的类型。
- `ValueType`：索引中每个值的类型。在实践中，这将是一个 64 位的 `RID`。
- `KeyComparator`：一个用于比较两个 `KeyType` 实例是否小于、大于或等于彼此的类。这些将包含在 `KeyType` 的实现文件中。

### 任务 #3 - 索引迭代器 (Task #3 - Index Iterator)

在你实现并彻底测试了 Task #1 和 Task #2 中的 B+ 树之后，你必须添加一个 C++ 迭代器，它能够高效地支持索引中条目的**有序扫描 (in-order scan)**。

基本思路是存储**兄弟页面指针 (sibling pointers)**，以便你可以高效地遍历叶子页面，然后实现一个迭代器来按顺序遍历索引中的每个键值对。

> ⚠️ **重要提示 (Fall 2025 新要求)：** 该迭代器**必须遵守墓碑机制**，因此应当**跳过**任何带有对应墓碑的键值对。

你的迭代器必须是 C++17 风格的 Iterator，至少包含以下方法：

- `isEnd()`：返回该迭代器是否指向最后一个键/值对。
- `operator++()`：移动到下一个键/值对。
- `operator*()`：返回该迭代器当前指向的键/值对。
- `operator==()`：返回两个迭代器是否相等。
- `operator!=()`：返回两个迭代器是否不相等。

你的 `BPlusTree` 也必须正确实现 `begin()` 和 `end()` 方法，以支持索引上的 C++ for-each 循环功能。

你必须且只能通过修改其头文件 (`src/include/storage/index/index_iterator.h`) 和对应的源文件 (`src/storage/index/index_iterator.cpp`) 来实现索引迭代器。

### 任务 #4 - 并发控制 (Task #4 - Concurrency Control)

在最后一个任务中，你将修改你的 B+ 树实现，使其安全地支持并发操作。你应该使用课堂上和教科书中描述的**乐观锁耦合/锁螃蟹 (optimistic latch coupling/crabbing)** 技术。

遍历索引的线程应该根据需要获取 B+ 树页面上的锁（latches），以确保安全的并发操作，并在安全时尽早释放父页面上的锁。

> ℹ️ **注意：** 绝不能在单个线程中获取两次相同的读锁。这可能会导致死锁。

### 开发路线图 (Development Roadmap)

构建 B+ 树索引有多种方法。此路线图仅作为粗略的概念指南，它基于教科书中概述的算法。即使不严格遵守该路线图，你也完全可以构建出能够通过我们所有测试的、语义正确的 B+ 树。选择完全取决于你。

   - *自我检查*：节点有哪些不同类型？键值是否可以插入到所有这些节点中？
2. **简单搜索 (Simple Search)**：给定一个键 K，在树上定义一个搜索机制来确定该键是否存在。
   - *自我检查*：键可以存在于多个节点中吗？这些键都相同吗？
3. **简单拆分 (Simple Splits)**：给定一个键 K 和一个已满的目标叶节点 L，将该键插入树中，同时保持树的一致性。
   - *自我检查*：你何时选择分裂一个节点？如何定义分裂？
4. **多重拆分 (Multiple Splits)**：在已满的叶节点 L 且其父节点 M 也已满的情况下，为键 K 定义插入。
   - *自我检查*：如果 M 的父节点也满了会发生什么？
5. **简单删除 (Simple Deletes)**：给定一个键 K 和一个至少半满的目标叶节点 L，从 L 中删除 K。

6. **简单合并 (Simple Coalesces)**：定义在删除操作后，对少于半满的叶节点 L 的键 K 的删除。

7. **复杂合并 (Not-So-Simple Coalesces)**：为节点 L 上不包含合适合并节点的键 K 定义删除。
   - *自我检查*：合并行为是否根据节点类型的不同而有所变化？

### 要求与提示 (Requirements and Hints)

- **不允许使用全局锁**来保护你的数据结构；你的实现必须支持合理的并发级别。换句话说，你不能锁住整个索引，并在操作完成时才解锁。我们建议你使用页面守卫类 `ReadPageGuard` 和 `WritePageGuard` 来为你的 B+ 树实现线程安全。如果你正确使用这些结构，你可以在这个项目上获得满分。
- 你可以向你的实现中添加函数，只要你保留所有我们原始的公共接口以用于测试目的。
- **不要使用 `malloc` 或 `new`** 为你的树分配大块内存。如果你需要为树创建新节点，或者在某些操作中需要缓冲区，你应该使用缓冲池管理器 (Buffer Pool Manager)。
- 在遍历内部或叶节点时，**必须使用二分查找 (binary search)** 来寻找插入值的位置。否则，你的实现大概率会在 Gradescope 上超时。
- 我们建议（但不强制要求）你在实现分裂时遵循以下规则：
  - **叶子节点**：在插入**之后**，当元素数量达到 `max_size` 时进行分裂。
  - **内部节点**：在插入**之前**，当元素数量达到 `max_size` 时进行分裂。
  - 这将确保对叶节点的插入绝不会在执行类似 `InsertIntoLeaf` 然后重新分配的操作时导致页面数据溢出；它还可以防止产生只有一个子节点的内部页面。当然，你也可以通过延迟分裂叶节点/处理只有一个子节点的内部节点等方式来实现。这完全取决于你，我们的测试用例不会对这些具体边界情况做强制测试。

#### 常见坑点 (Common Pitfalls)

- 我们不会测试你的迭代器是否支持线程安全的叶子扫描。然而，一个正确的实现需要 Leaf Page 在无法获取其兄弟页面上的锁时抛出 `std::exception`，以避免潜在的死锁。
- 如果你正确地实现了一个并发的 B+ 树索引，每个线程将总是**从头部页面 (header page) 向下**获取锁。当你释放锁时，请确保你以相同的顺序（从头部页面向下）释放它们。
- 在实现 Page 类（Task 1）时，确保只添加**平凡构造类型**（例如 `int`）的类字段。**不要添加 vector**，不要修改 `key_array_` 和 `value_array_`。

## 第二部分：P2 深度技术解析与保姆级实现指南

### 一、 核心架构设计

### 二、 2025秋季学期最关键的变化：Tombstone (墓碑缓冲区) 算法细节

这是本学期最具挑战性也是最容易出错的革新。为了缓解高频物理删除带来的页数据移动开销，B+ 树的叶子页面引入了类似于 $B^\epsilon$-tree 的 **Tombstone 缓冲区**。

#### 1. 什么是 Tombstone Buffer？

在叶子页面物理结构中，包含一个最大容量为 $k$ (由参数 `NumTombs` 控制) 的缓冲区，用来记录“已删除但尚未真正物理清除”的键的**物理数组索引 (physical indices)**。

#### 2. 删除 (Remove) 的流程式控制制 ($k > 0$)

当调用 `Remove(key)` 时，我们定位到叶节点，并通过二分查找找到物理数组中的键，其物理索引为 $idx$：

1. **如果** $k = 0$：退化为传统的 B+ 树，直接执行物理删除，移动数组元素，`size_` 减 1。
2. **如果** $k > 0$：
   - 不执行物理删除，不移动元素。
   - 将该索引 $idx$ **追加 (push_back)** 到该 Leaf Page 的墓碑缓冲区。
   - **检查缓冲区大小是否达到了** $k$：
     - 若未达 $k$，删除操作在此结束，直接返回。
     - 若达到了 $k$，则**触发物理驱逐**：
       - 取出墓碑缓冲区中**最老的一个墓碑**（即缓冲区头部的索引 $idx_{old}$）。
       - 执行物理删除：在物理数组中将 $idx_{old}$ 处的元素删除，使该索引后面的所有元素全部向左移动一位，并将 `size_` 减 1。
       - **同步更新缓冲区中其余墓碑的索引**：因为元素左移了，任何记录在缓冲区中且**物理索引大于** $idx_{old}$ **的墓碑，其记录的物理索引必须全部减 1**！
       - 将最老墓碑从缓冲区中移出 (pop_front)。

##### ⚠️ 极易发生的 Bug 举例

假设物理数组为 `[A, B, C, D]`，限制 $k = 2$。

1. 删 B（索引 1）：Buffer 变成 `[1]`。
2. 删 D（索引 3）：Buffer 变成 `[1, 3]`。此时 Buffer 大小达 $k=2$，触发驱逐。
3. 最老墓碑为 `1` (即B)。物理删除索引 1，物理数组变成 `[A, C, D]`。
4. **关键更新**：原本 D 的物理索引是 `3`，由于 B 被删除，D 移到了索引 `2`。此时 Buffer 中记录 D 的索引必须同步变更为 `2`。Buffer 最终变为 `[2]`。

#### 3. 查找 (GetValue)

在叶节点上进行二分查找。如果找到了物理索引 $idx$：

- **必须检查** $idx$ **是否存在于当前页面的墓碑缓冲区中**。
- 如果存在，说明该键在逻辑上已被删除。`GetValue` 应该返回 `false`（相当于未找到）。
- 如果不存在，返回该值，并返回 `true`。

#### 4. 插入 (Insert) 冲突判定

由于墓碑的存在，“逻辑不存在”和“物理不存在”被剥离开。 当插入一个键 `key` 时：

- 如果物理数组中不存在该 `key`：执行标准物理插入。
- 如果物理数组中存在该 `key`，但**它在墓碑缓冲区中**（即逻辑上已删除）：
  - 这不应该判定为“唯一键冲突”。你应当“复活”该物理条目：用新值覆盖该物理位置的值，并将该索引从墓碑缓冲区中移除。
  - 注意：移除该墓碑可能会导致其余墓碑不需要做任何位移调整，因为没有发生物理数组的压缩移动。
- 如果物理数组中存在且**不在墓碑中**：触发唯一键冲突，直接返回 `false`。

#### 5. 分裂 (Split) 与合并 (Coalesce) 时的墓碑转移

当对 Leaf Page 进行 Split、Merge 或 Redistribution 时：

- **Split (分裂)**：
  - 右半部分被拷贝到新的 Leaf Page。
  - 原本属于右半部分元素（物理索引 $\ge mid$）的墓碑必须随之迁移到新页面。在新页面中，它们的物理索引需要平移更新（新索引 = 原索引 - $mid$）。
  - 留在原页面的墓碑索引保持不变。
- **Merge (合并 Leaf B 合并入 Leaf A)**：
  - 将 B 的所有物理元素追加到 A 后面。
  - B 的所有墓碑也被转移到 A 的缓冲区。B 墓碑在 A 中的新物理索引 = B 原物理索引 + A 合并前的 `size_`。
  - **墓碑的时间顺序处理**：根据规范，“被并入的目标节点（A），其原有的墓碑应该被视为更老”。所以，B 转移过来的墓碑应当**追加在 A 现有墓碑缓冲区之后（更年轻）**。
  - 如果合并后 A 的墓碑缓冲区大小超过了 $k$，则需要不断循环触发物理驱逐，直到缓冲区大小 $\le k$。

### 三、 各 Task 落地开发指南

#### Task #1: B+Tree Pages

主要实现底层的辅助函数，务必保证它们在单测中 100% 准确。

1. **B+Tree Page (Base)**:
   - 实现 `GetPageType`, `SetPageType`, `GetSize`, `SetSize`, `GetMaxSize`, `SetMaxSize` 等 getter/setter。
2. **Internal Page**:
   - 第一个 Key 是无效的 (invalid)，查找和比较时要从 index 1 开始。
   - 提供方便的辅助函数，如：`Lookup(key, comparator)`，通过二分查找确定该去哪个子节点。
3. **Leaf Page**:
   - 提供墓碑相关的辅助函数：`GetTombstones()`, `IsTombstoned(idx)` 等。
   - 物理插入 `Insert(key, value)` 和 物理删除。

#### Task #2: B+Tree Operations

1. **查找 (`GetValue`)**:
   - 从 `root_page_id` 开始，利用 `ReadPageGuard` 获取读锁，一路向下搜寻内部节点。
   - 每次利用 `Lookup` 定位到下一层子节点的 `page_id`，换锁（Latch Crabbing，见 Task 4），直到叶节点。
   - 在叶节点内搜索，注意判定墓碑，返回结果。
2. **插入 (`Insert`)**:
   - 获取 `header_page_` 的写锁守卫以防根节点分裂。
   - 向下寻找目标叶节点，记录访问路径于 `Context` 中。
   - 如果叶节点满，触发分裂：
     - 分配新页面，创建新叶子节点。
     - 拷贝半数数据及对应的墓碑。
     - 将新页面的 `page_id` 和分割键 (split key) 插入到父节点中（若父节点也满，递归分裂父节点）。
3. **删除 (`Remove`)**:
   - 定位到叶节点，应用 Tombstone 规则。
   - 若没有墓碑限制或墓碑驱逐导致物理 size 小于 `max_size / 2`（或者对于叶节点，小于 `GetMinSize()`），则可能需要与其物理相邻的兄弟节点进行**合并 (Merge/Coalesce)** 或 **重新分配 (Redistribute)**。
   - 合并时注意更新父节点中的键。如果合并导致根节点失去所有键，更新根节点为唯一的子节点，或者直接变为空树。

#### Task #3: Index Iterator

- 迭代器初始化时，定位到最左侧的叶子节点，并指向物理索引 0。

- 在 `operator++` 移动时，必须在一个循环中寻找下一个有效条目：

  ```
  void Advance() {
      curr_idx_++;
      while (curr_page_id != INVALID_PAGE_ID) {
          auto leaf_page = reinterpret_cast<LeafPage*>(page->GetData());
          if (curr_idx_ < leaf_page->GetSize()) {
              if (!leaf_page->IsTombstoned(curr_idx_)) {
                  // 找到了一个没有被标记为墓碑的有效键值对，停止
                  break; 
              }
              curr_idx_++; // 遇到了墓碑，跳过，继续向后看
          } else {
              // 当前页读完了，通过 next_page_id 跨越到右侧兄弟节点
              auto next_page_id = leaf_page->GetNextPageId();
              // 释放当前页，锁上下一页，将 curr_idx_ 重置为 0 ...
              curr_idx_ = 0;
          }
      }
  }
  ```

#### Task #4: Concurrency Control (锁螃蟹技术)

锁螃蟹技术核心是：**在可以保证安全的前提下，尽早释放祖先节点的锁**。

```
           [Header Page] (Write Latch)
                 |
                 v
            [Parent Page] (Write Latch)  <-- 若判定当前操作不会导致 Parent 发生 Split/Merge，
                 |                           则在此处安全释放 Header 的锁！
                 v
            [Child Page] (Write Latch)   <-- 若判定 Child 安全，释放 Parent 锁。
```

- **对于 Read 操作**：
  - 从 Root 开始向下，先锁子节点，再释放父节点的读锁。
- **对于 Write 操作（Insert/Delete）**：
  - 获取 Header 页面和 Root 页面的写锁。
  - 在向下遍历时，获取子页面的写锁。
  - **判定安全性**：
    - **插入时**：如果子节点未满（即 $size < max\_size$），即使发生插入也不会引发分裂。此时，该子节点之上的所有祖先节点的写锁都可以安全释放（从 Context 中 Drop 掉它们）。
    - **删除时**：如果子节点在删除/驱逐后仍大于 $min\_size$（即 $size > min\_size$），即使发生删除也不会引发合并。此时，可以安全释放祖先节点的所有写锁。

### 四、 本地开发与高效调试步骤

1. **严禁全局锁**：Gradescope 的并发测试会严查并发度。如果使用一把大锁保平安，你会被判 0 分。
2. **善用可视化工具**：在编写 Task 2 时，多用 `b_plus_tree_printer` 导出 `.dot` 文件，对照在线可视化工具观察树的节点分裂、合并、键的分配是否符合 B+ 树的定义。
3. **使用 ASAN/LSAN 严查内存泄漏**：在 CMake 配置中开启 Debug 模式编译并运行测试，任何没有正确 `unpin` 页面或没有丢弃 `PageGuard` 的行为都会导致严重的内存泄漏。
4. **死锁排查**：在多线程并发测试中，如果发生挂起超时，大概率是因为在向下遍历时存在“向上获取锁”或“重复获取同一页面锁”的行为。务必保证加锁顺序严格**自上而下**。
