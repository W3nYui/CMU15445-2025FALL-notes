BusTub 支持 `EXPLAIN` 命令来打印查询的执行计划。 例如：`EXPLAIN SELECT * FROM __mock_table_1;`

`EXPLAIN` 的结果提供了查询处理层内部转换过程的概览：

2. **Planner (计划器)**: 生成查询计划树（数据从叶子流向根）。
3. **Optimizer (优化器)**: 优化查询计划（例如移除冗余的 Projection）。

### 项目规范 (Project Specification)

在这个项目中，你将向 BusTub 添加新的算子执行器和查询优化。 BusTub 使用**基于行的向量化查询处理模型 (Row-based vectorization query processing model)**，其中每个执行器都实现了一个 `Next` 函数，用于获取最多包含 `BUSTUB_BATCH_SIZE` 个元组的**元组批次 (Tuple Batch)**。

当 DBMS 调用执行器的 `Next` 函数时，执行器返回：

1. 一批元组 (a batch of some tuples)
2. 或者一个指示没有更多元组的标志。

在这种方法下，每个执行器都实现了一个循环，不断对其子节点调用 `Next` 以获取元组，并按批次处理它们。在 BusTub 的实现中，每个执行器的 `Next` 函数除了返回元组批次外，还返回一批**记录标识符 (RID)**。

所有的测试用例都是使用 `SQLLogicTest` 格式编写的。

### 任务 #1 - 访问方法执行器 (Task #1 - Access Method Executors)

在此任务中，你需要实现从存储系统中读取和写入表的执行器。

1. **SeqScan (顺序扫描)**: 迭代遍历表，并每次返回一个批次的元组。 *提示*: 使用 `TableIterator` 时，注意前缀递增和后缀递增的区别。不要输出被标记为删除的元组（检查 `TupleMeta` 的 `is_deleted_` 字段）。输出是匹配元组的副本及其原始 RID。
2. **Insert (插入)**: 将元组插入表中并更新任何受影响的索引。它有且只有一个子节点提供值。执行器应当输出一个单行整型元组，指示插入的行数。
3. **Update (更新)**: 修改指定表中现有的元组。输出修改的行数。注意同步更新索引。 *提示*: 要实现更新，首先删除受影响的元组，然后插入一个新元组。
4. **Delete (删除)**: 从表中逻辑删除记录。输出删除的行数，并更新索引。
5. **IndexScan (索引扫描) & 优化器规则**: 你需要实现 `SeqScanAsIndexScan` 优化器规则（谓词下推）。当谓词（Predicate）中包含被索引列的等值测试时，将 `SeqScanPlanNode` 转换为 `IndexScanPlanNode`。需要支持单点查找和同一索引的多个点查找（`OR` 条件）。

### 任务 #2 - 聚合与连接执行器 (Task #2 - Aggregation & Join Executors)

1. **Aggregation (聚合)**: 为每一组输入计算聚合函数。输出 Schema 为 Group-by 列 + 聚合列。 在本项目中，你可以假设聚合哈希表**完全可以装入内存**。我们提供了一个 `SimpleAggregationHashTable` 数据结构，你需要完成其 `CombineAggregateValues` 函数。 *提示*: 聚合是一个 Pipeline Breaker（管道断路器）。需要仔细权衡是在 `Init()` 还是 `Next()` 中执行聚合的 Build（构建）阶段。必须处理聚合输入中的 NULL 值。
2. **NestedLoopJoin (嵌套循环连接)**: 针对没有索引的表进行连接，实现内连接或左外连接。
3. **NestedIndexJoin (嵌套索引连接)**: 当连接条件的右侧有索引时，通过探测索引进行高效连接。

### 任务 #3 - 哈希连接执行器与优化 (Task #3 - HashJoin Executor and Optimization)

如果等值连接条件没有索引，NestedLoopJoin 的性能会极差。你需要实现：

1. **HashJoin (哈希连接)**: 在内存中构建哈希表以执行等值连接。Build 阶段读取右侧子节点的所有元组构建哈希表，Probe 阶段读取左侧元组进行探测。
2. **HashJoin Optimization (优化器规则)**: 实现规则将 `NestedLoopJoin` 转换为 `HashJoin`（当检测到等值条件 `=` 时）。

### 任务 #4 - 外部归并排序 + 限制执行器 + 窗口函数

实现 `SortExecutor`（内存排序）、`LimitExecutor`（截断输出批次）和 `WindowFunctionExecutor`（窗口函数，支持 `PARTITION BY` 和 `ORDER BY` 进行累计计算）。

## 第二部分：P3 深度技术解析与保姆级实现指南

### 一、 核心架构设计：向量化执行模型 (Vectorization Model)

**⚠️ Fall 2025 赛季最大的变化与难点：** 传统的 Volcano 模型中，`Next()` 每次只返回一个 `Tuple`。但在当前的 Vectorization 模型中，`Next()` 每次返回的是一个 `std::vector<Tuple>`（批次），其最大容量为 `BUSTUB_BATCH_SIZE`。

**带来的技术挑战：状态保留 (State Retention)** 如果一个算子（如 Filter, Join）在处理子节点传来的一个 Batch 时，自身的输出 Batch 已经达到了 `BUSTUB_BATCH_SIZE`，那么**它必须暂停处理**并返回。当下次调用 `Next()` 时，它必须**准确地从上次中断的地方（某个子节点的某个 Batch 的某个 Index 处）继续处理**。

你必须在所有的 Executor 类内部定义成员变量（例如 `current_batch_`, `current_idx_`）来显式地保存这种中间状态。

### 二、 Pipeline 构建逻辑：Breaker vs Pipeline

根据数据处理的时机，算子可以分为两类，这决定了你的 `Init` 和 `Next` 应该怎么写：

1. **流水线算子 (Pipeline Operators)**：如 SeqScan, Filter, Limit, NLJ。
   - `Init()`：仅调用子节点的 `Init()`，重置自身的状态变量。
   - `Next()`：按需调用子节点的 `Next()` 获取一个批次，处理后立刻输出。
2. **管道阻断算子 (Pipeline Breakers)**：如 Aggregation, HashJoin, Sort。
   - `Init()`：不仅要调用子节点的 `Init()`，**还要在一个 `while` 循环中抽干（Exhaust）子节点的所有数据**，并在内存中构建完整的数据结构（Hash表、排序数组等）。
   - `Next()`：不再调用子节点，而是基于在 `Init` 中构建好的数据结构，按批次输出（即维护一个 Iterator 或 Index，每次取 `BUSTUB_BATCH_SIZE` 个返回）。

### 三、 各 Task 落地开发指南 (包含易错点)

#### Task 1: 访问方法与单点修改

**1. SeqScanExecutor**

- 内部维护一个 `TableIterator`。
- 在 `Next()` 中，使用 `while` 循环读取记录，跳过 `tuple_meta.is_deleted_ == true` 的记录。
- 将合法的 Tuple 追加到输出 batch 中，当 batch 满时 `return true`；如果迭代器到头了且 batch 非空则返回 true，否则返回 false。

**2. Insert/Update/Delete (DML Executors)**

- **易错点 1**：DML 语句本身虽然修改了数据，但在当前 SQL 语义下，它们需要向下游（往往就是最终输出）返回一个且仅一个包含整数的 Tuple（即受影响的行数）。
- **易错点 2**：即便子节点的元组分多个 Batch 传来，DML 执行器的 `Next()` 也应该**只在第一次被调用时返回一行结果**。如果被调用第二次，必须返回 `false`。
- **Index 的同步**：对于 Insert，调用 `table_info->table_->InsertTuple` 拿到新 RID 后，必须遍历 `catalog_->GetTableIndexes()`，通过 `index->InsertEntry()` 更新所有索引。Update 则是先 Delete 原 Entry，再 Insert 新 Entry。

**3. SeqScanAsIndexScan 优化规则**

- 检查 `plan->GetType() == PlanType::Filter`，且其子节点为 `SeqScan`。
- 解析 Filter 中的 `EqualExpression`，检查涉及的列是否有对应的 Index。
- 如果有，并且是单点查询或 `OR` 查询的合取，用 `IndexScanPlanNode` 替换这部分树。

#### Task 2: 聚合与嵌套连接

**1. Aggregation (Hash 聚合)**

- **Init() 中**：实例化 Hash 表，通过一个 `while` 循环不断调用 `child->Next()`。对接收到的每个 Tuple 的每个元素，提取 Group-By Key 和 Aggregate Value，调用 `aht_.InsertCombine(key, val)`。
- **特殊 Corner Case**：如果表为空（即子节点未返回任何批次）：
  - 如果查询包含 Group-By，则不应该输出任何结果（`Next` 直接返回 false）。
  - 如果查询**没有** Group-By（比如 `SELECT COUNT(*) FROM empty_table`），则**必须**输出一行默认值（如 COUNT 输出 0，MIN 输出 NULL）。
- **Next() 中**：从 `aht_.Begin()` 开始遍历，组装 Tuple 并按 Batch 返回。注意保存 Iterator 为类成员以应对多次调用。

**2. NestedLoopJoin (向量化陷阱重灾区)**

- **算法模型**：双重 `for` 循环。外层遍历 Left Batch，内层遍历 Right Batch。

- **状态维护**：

  ```
  // 必须在 Executor 中维护的成员状态：
  std::vector<Tuple> left_batch_;
  uint32_t left_idx_{0};
  std::vector<Tuple> right_batch_;
  uint32_t right_idx_{0};
  ```

- **左外连接 (Left Outer Join)**：对于左表的一条记录，如果在右表中完全没有匹配项，则必须用 NULL 填充右表的字段输出一条记录。你需要一个布尔标识 `left_matched_` 来记录当前左侧元组是否产生过任何连接。

#### Task 3: HashJoin

- **Init() 中构建阶段**：首先执行右子树的 `Init` 和抽干操作，将所有右表元组根据 Join Key 散列到 `std::unordered_map<HashKey, std::vector<Tuple>>` 中。
- **Next() 中探测阶段**：不断从左子树获取 Batch。对于 Batch 中的每一个 Tuple，计算 Hash Key 并去右表 Hash 表中查找匹配。
- **内存防爆**：注意由于是等值连接，一个左 Tuple 可能匹配多个右 Tuple，务必通过 `current_left_tuple_` 和 `current_right_match_idx_` 来处理一次匹配产出超过 `BUSTUB_BATCH_SIZE` 的情况。

#### Task 4: Sort, Limit, Window

**1. Sort**

- 作为一个 Pipeline Breaker，在 `Init()` 抽干子节点并将所有的 Tuple 保存在 `std::vector<Tuple>` 中。
- 使用 `std::sort` 对该 vector 进行排序。你需要自定义一个 Comparator 闭包，根据传入的 `OrderBy` 规则从左到右进行比较。注意处理升序/降序。

**2. Limit**

- 简单算子，维护一个 `count_`，每次从子节点的 Batch 中截取剩余需要的额度，当 `count_ == limit_` 时立刻返回，且后续 `Next` 永远返回 false。

**3. Window Function**

- 本质是**排序 + 分区 + 聚合**的结合体。
- 整个逻辑可以在 `Init()` 中完成处理（由于测试数据保证能放下）：
  1. 获取所有子节点的数据。
  2. 按照 `Partition By` 的字段 + `Order By` 的字段进行全局排序。
  3. 遍历排序好的数组。当发现 `Partition By` 字段变化时，重置内部聚合状态；否则在当前聚合状态上累加。
  4. 将每一行附加好对应的 Window 计算结果，缓存为完整的输出列表。
- `Next()` 仅负责将算好的列表按 Batch 切分返回。

### 结语

Project 3 的核心在于将上层 SQL 逻辑与底层的存储引擎和缓冲池桥接起来。在向量化引擎模型下，必须牢记**边界控制与中间状态保存**。建议在开发前先通过纸笔画出每个算子的状态转移图，这将极大减少后续的调试成本。
