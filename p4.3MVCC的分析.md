# BusTub Project 4 Task 3.4: Update & Delete Executor 学习指南

这份文档面向你当前仓库的进度：`SeqScanExecutor` 已经开始按 MVCC 读版本链，`InsertExecutor`
也已经把新 tuple 的 `TupleMeta::ts_` 设置为事务临时时间戳。接下来 `UpdateExecutor` 和
`DeleteExecutor` 不能再沿用 Project 3 的“直接改表”写法，而要维护 MVCC 版本链。

> 课程页面：<https://15445.courses.cs.cmu.edu/fall2025/project4/#task3>

## 1. 这题到底要做什么

Task 3.4 要你让 `UPDATE` 和 `DELETE` 成为 MVCC 写操作。它们不只是改变表里的 tuple，还必须：

- 检查写写冲突。
- 在当前事务第一次修改某个 RID 时创建 undo log。
- 在当前事务重复修改同一个 RID 时更新已有 undo log，而不是再追加一个。
- 用 `UpdateTupleAndUndoLink` 原子地更新 table heap 里的 base tuple 和 undo link。
- 把写过的 RID 记录进 transaction write set，方便 commit / abort / serializable verification。
- 正确维护索引项，尤其是 update 改到了索引列或 delete 删除了索引可见项时。
- 对 `UpdateExecutor` 做 pipeline breaker：先完整消费 child，再统一执行更新。

一句话版本：**Project 3 只关心“当前表长什么样”，Project 4 要关心“每个事务在自己的时间点应该看到什么”。**

## 2. 和一开始 update/delete 的差别

你当前的 `DeleteExecutor` 大致是：

```cpp
table_info_->table_->UpdateTupleMeta(TupleMeta{0, true}, rid);
index->DeleteEntry(old_key, rid, txn);
```

你当前的 `UpdateExecutor` 大致是：

```cpp
UpdateTupleMeta(TupleMeta{0, true}, old_rid);
DeleteEntry(old_key, old_rid, txn);
InsertTuple(TupleMeta{0, false}, new_tuple, lock_mgr, txn, table_oid);
InsertEntry(new_key, new_rid, txn);
```

这在 Project 3 可以，因为那时没有版本可见性。Project 4 里这样写会有几个问题：

- `UPDATE` 不应该 delete old RID 再 insert new RID。MVCC 版本链是挂在同一个 RID 上的，更新应该原地改 base tuple。
- `TupleMeta::ts_` 不能写成 0。当前事务未提交的写入要写成 `txn->GetTransactionTempTs()`。
- 旧版本不能丢。旧值要放进 undo log，给旧事务按 read timestamp 重建历史版本。
- 写写冲突不能沉默覆盖。遇到其他事务已经写过、且这个写入对当前事务不可见时，要 `SetTainted()` 并抛 `ExecutionException`。
- 同一个事务多次 update/delete 同一个 RID，最多只应该有一个 undo log。后续修改要合并进第一次的 undo log。
- Update 要避免边扫描边改导致的 Halloween 类问题，所以它是 pipeline breaker。

## 3. 这个 task 的目的是什么

这个任务不是为了“让 UPDATE/DELETE 能跑”，它真正训练的是 MVCC 写路径：

- 读路径已经可以通过 `CollectUndoLogs` + `ReconstructTuple` 找到历史版本。
- 写路径现在要负责制造这些历史版本。
- commit 只需要把自己写过的 tuple 的临时时间戳改成 commit timestamp。
- abort 以后要能通过 undo log 恢复旧状态。
- 后续索引、主键、并发测试都会依赖这一层。

所以这题的核心心智模型是：

```text
table heap 里永远放“最新版本”
undo log 链里放“怎么退回旧版本”
read_ts 决定读者要把最新版本往回退到哪里
```

## 4. 推荐先补的 helper

不要把所有逻辑塞进 `UpdateExecutor::Next` 和 `DeleteExecutor::Next`。这两个 executor 的大部分流程是共享的，建议在
`execution_common.h/.cpp` 里补几个小 helper。

### 4.1 判断一个 tuple 是否被其他事务写过

用途：Update/Delete 在真正写入前调用。

推荐语义：

```cpp
auto IsWriteWriteConflict(const TupleMeta &meta, Transaction *txn) -> bool;
```

判断规则：

```text
如果 meta.ts_ == txn->GetTransactionTempTs()
  这是当前事务自己写的，不冲突。

如果 meta.ts_ 是临时时间戳，并且不是当前事务的 temp ts
  其他未提交事务写过，冲突。

如果 meta.ts_ > txn->GetReadTs()
  当前事务开始之后，有别的事务提交了新版本，冲突。

其他情况
  当前事务可以写。
```

骨架：

```cpp
auto IsTempTs(timestamp_t ts) -> bool {
  return ts >= TXN_START_ID;
}

auto IsSelfModification(const TupleMeta &meta, Transaction *txn) -> bool {
  return meta.ts_ == txn->GetTransactionTempTs();
}

auto IsWriteWriteConflict(const TupleMeta &meta, Transaction *txn) -> bool {
  if (IsSelfModification(meta, txn)) {
    return false;
  }
  if (IsTempTs(meta.ts_)) {
    return true;
  }
  return meta.ts_ > txn->GetReadTs();
}
```

### 4.2 冲突时统一 taint 并抛异常

测试里有提示：遇到写写冲突时，既要 `SetTainted()`，也要抛 `ExecutionException`。

推荐 helper：

```cpp
[[noreturn]] void TaintAndThrow(Transaction *txn, const std::string &msg) {
  txn->SetTainted();
  throw ExecutionException(msg);
}
```

需要 include：

```cpp
#include "common/exception.h"
```

### 4.3 计算 update 的目标 tuple

用途：把 `UpdatePlanNode::target_expressions_` 单独抽出来，避免主流程变长。

推荐签名：

```cpp
auto MakeUpdatedTuple(const Schema &table_schema,
                      const Schema &child_schema,
                      const Tuple &old_tuple,
                      const UpdatePlanNode *plan) -> Tuple;
```

骨架：

```cpp
auto MakeUpdatedTuple(const Schema &table_schema,
                      const Schema &child_schema,
                      const Tuple &old_tuple,
                      const UpdatePlanNode *plan) -> Tuple {
  std::vector<Value> values;
  values.reserve(table_schema.GetColumnCount());

  for (uint32_t col = 0; col < table_schema.GetColumnCount(); col++) {
    values.push_back(plan->target_expressions_[col]->Evaluate(&old_tuple, child_schema));
  }

  return {values, &table_schema};
}
```

### 4.4 判断 tuple 内容是否真的变化

你仓库里已经有 `IsTupleContentEqual` 在用。如果 update 后的新 tuple 和旧 tuple 完全一样，可以选择跳过写入。

推荐规则：

```text
UPDATE table SET a = a
不应该生成 undo log，也不应该改 ts。
```

骨架：

```cpp
if (IsTupleContentEqual(old_tuple, new_tuple)) {
  continue;
}
```

### 4.5 封装 index 更新

Update/Delete 最容易把索引写乱。建议拆两个 helper：

```cpp
void DeleteIndexEntries(const std::vector<std::shared_ptr<IndexInfo>> &indexes,
                        const Schema &table_schema,
                        const Tuple &old_tuple,
                        RID rid,
                        Transaction *txn);

void InsertIndexEntries(const std::vector<std::shared_ptr<IndexInfo>> &indexes,
                        const Schema &table_schema,
                        const Tuple &new_tuple,
                        RID rid,
                        Transaction *txn);
```

delete 的时候：

```cpp
DeleteIndexEntries(indexes, table_schema, old_tuple, rid, txn);
```

update 的时候：

```cpp
if (!IsTupleContentEqual(old_tuple, new_tuple)) {
  DeleteIndexEntries(indexes, table_schema, old_tuple, rid, txn);
  InsertIndexEntries(indexes, table_schema, new_tuple, rid, txn);
}
```

更细一点可以只在 key 变化时更新对应索引：

```cpp
auto old_key = old_tuple.KeyFromTuple(table_schema, index_info->key_schema_, index_info->index_->GetKeyAttrs());
auto new_key = new_tuple.KeyFromTuple(table_schema, index_info->key_schema_, index_info->index_->GetKeyAttrs());
if (!IsTupleContentEqual(old_key, new_key)) {
  index->DeleteEntry(old_key, rid, txn);
  index->InsertEntry(new_key, rid, txn);
}
```

刚开始可以先用“删旧 key、插新 key”的保守版本，理解清楚后再优化。

### 4.6 统一执行一次 MVCC modify

这是最重要的 helper。Update 和 Delete 最后都可以变成：

```text
给定 rid + old_tuple + target_tuple/nullptr
检查冲突
生成或更新 undo log
原子更新 table tuple + undo link
记录 write set
更新索引
```

推荐签名：

```cpp
auto ModifyTupleForMvcc(TransactionManager *txn_mgr,
                        Transaction *txn,
                        TableInfo *table_info,
                        RID rid,
                        const Tuple &visible_old_tuple,
                        const Tuple *target_tuple,
                        bool is_delete) -> bool;
```

不过你也可以不要做这么大的 helper，只拆出更小的：

```cpp
auto BuildUndoLinkForFirstWrite(const Schema &schema, const TupleMeta &base_meta, const Tuple &base_tuple,
                                const Tuple *target_tuple, std::optional<UndoLink> prev_link,
                                Transaction *txn) -> std::optional<UndoLink>;

auto RefreshUndoLogForSelfWrite(const Schema &schema, const TupleMeta &base_meta, const Tuple &base_tuple,
                                const Tuple *target_tuple, std::optional<UndoLink> self_link,
                                Transaction *txn) -> void;

auto ApplyTupleMutationAtomically(TransactionManager *txn_mgr, TableHeap *table_heap, Transaction *txn, RID rid,
                                  std::optional<UndoLink> undo_link, const TupleMeta &new_meta,
                                  const Tuple &new_tuple) -> bool;
```

我的建议：**先做小 helper，等 Update 和 Delete 都跑通后，再考虑合并。**

## 5. Undo log 的两条路径

每个 RID 对当前事务有两种情况。

### 5.1 第一次修改这个 RID

特征：

```cpp
base_meta.ts_ != txn->GetTransactionTempTs()
```

此时要新建 undo log：

```cpp
auto new_log = GenerateNewUndoLog(
    &table_info->schema_,
    &base_tuple,
    target_tuple_or_null,
    base_meta.ts_,
    prev_undo_link.value_or(UndoLink{}));

auto new_link = txn->AppendUndoLog(new_log);
```

然后把 table heap 的 undo link 更新成 `new_link`。

注意：`base_tuple` 应该是 table heap 当前最新版本，不一定等于 child 给你的 visible tuple。child tuple 是按当前事务视角读出来的，table heap tuple 是版本链头部。

### 5.2 当前事务已经修改过这个 RID

特征：

```cpp
base_meta.ts_ == txn->GetTransactionTempTs()
```

此时不要 append 新 undo log。要拿当前 RID 的 undo link，找到自己事务里的旧 log，然后修改它：

```cpp
auto self_link = current_undo_link.value();
auto old_log = txn->GetUndoLog(self_link.prev_log_idx_);

auto updated_log = GenerateUpdatedUndoLog(
    &table_info->schema_,
    base_meta.is_deleted_ ? nullptr : &base_tuple,
    target_tuple_or_null,
    old_log);

txn->ModifyUndoLog(self_link.prev_log_idx_, updated_log);
```

关键点：

- 一个事务对同一个 RID 最多一个 undo log。
- 这个 undo log 永远记录“第一次修改前”的状态。
- 第二次、第三次 update/delete 只是把这个 log 合并得更准确。
- 如果这个 RID 是当前事务刚 insert 的，通常没有 undo log，后续 update/delete 不需要创建历史版本给别人看，因为别人本来就看不到这条未提交插入。

## 6. DeleteExecutor 的代码骨架

文件：`src/execution/delete_executor.cpp`

目标：不要再 `UpdateTupleMeta(TupleMeta{0, true})`。要在同一个 RID 上写 delete marker，并维护 undo log。

```cpp
auto DeleteExecutor::Next(std::vector<Tuple> *tuple_batch,
                          std::vector<RID> *rid_batch,
                          size_t batch_size) -> bool {
  tuple_batch->clear();
  rid_batch->clear();

  if (has_deleted_) {
    return false;
  }

  auto txn = exec_ctx_->GetTransaction();
  auto txn_mgr = exec_ctx_->GetTransactionManager();
  auto indexes = exec_ctx_->GetCatalog()->GetTableIndexes(table_info_->name_);
  auto table_heap = table_info_->table_.get();
  const auto &schema = table_info_->schema_;

  int32_t delete_cnt = 0;
  std::vector<Tuple> child_tuples;
  std::vector<RID> child_rids;

  while (child_executor_->Next(&child_tuples, &child_rids, batch_size)) {
    for (size_t i = 0; i < child_tuples.size(); i++) {
      const auto &visible_tuple = child_tuples[i];
      RID rid = child_rids[i];

      auto [base_meta, base_tuple, undo_link] =
          GetTupleAndUndoLink(txn_mgr, table_heap, rid);

      if (IsWriteWriteConflict(base_meta, txn)) {
        TaintAndThrow(txn, "write-write conflict in delete");
      }

      if (base_meta.is_deleted_ && base_meta.ts_ == txn->GetTransactionTempTs()) {
        // 自己已经删过了，不重复计数。
        continue;
      }

      TupleMeta new_meta{txn->GetTransactionTempTs(), true};
      Tuple new_base_tuple = base_tuple;
      std::optional<UndoLink> new_undo_link = undo_link;

      if (base_meta.ts_ == txn->GetTransactionTempTs()) {
        // 当前事务已经写过这个 RID：更新已有 undo log。
        // 如果 undo_link 不存在，说明这是本事务 insert 后又 delete，可以不生成 undo log。
        if (undo_link.has_value() && undo_link->IsValid() &&
            undo_link->prev_txn_ == txn->GetTransactionId()) {
          auto old_log = txn->GetUndoLog(undo_link->prev_log_idx_);
          auto updated_log = GenerateUpdatedUndoLog(
              &schema,
              base_meta.is_deleted_ ? nullptr : &base_tuple,
              nullptr,
              old_log);
          txn->ModifyUndoLog(undo_link->prev_log_idx_, updated_log);
        }
      } else {
        // 当前事务第一次写这个 RID：追加新 undo log。
        auto log = GenerateNewUndoLog(
            &schema,
            base_meta.is_deleted_ ? nullptr : &base_tuple,
            nullptr,
            base_meta.ts_,
            undo_link.value_or(UndoLink{}));
        new_undo_link = txn->AppendUndoLog(log);
      }

      bool ok = UpdateTupleAndUndoLink(
          txn_mgr,
          rid,
          new_undo_link,
          table_heap,
          txn,
          new_meta,
          new_base_tuple,
          [&](const TupleMeta &latest_meta, const Tuple &latest_tuple,
              RID latest_rid, std::optional<UndoLink> latest_link) {
            return !IsWriteWriteConflict(latest_meta, txn);
          });

      if (!ok) {
        TaintAndThrow(txn, "write-write conflict in delete");
      }

      DeleteIndexEntries(indexes, schema, visible_tuple, rid, txn);
      txn->AppendWriteSet(table_info_->oid_, rid);
      delete_cnt++;
    }
  }

  tuple_batch->emplace_back(
      std::vector<Value>{ValueFactory::GetIntegerValue(delete_cnt)},
      &GetOutputSchema());

  has_deleted_ = true;
  return true;
}
```

## 7. UpdateExecutor 的代码骨架

文件：`src/execution/update_executor.cpp`

目标：不要 delete old RID 再 insert new RID。Update 应该在同一个 RID 上把 base tuple 改成新 tuple。

额外注意：Update 是 pipeline breaker。先把 child 的输出完整收集起来，再逐条修改。

```cpp
auto UpdateExecutor::Next(std::vector<Tuple> *tuple_batch,
                          std::vector<RID> *rid_batch,
                          size_t batch_size) -> bool {
  tuple_batch->clear();
  rid_batch->clear();

  if (has_updated_) {
    return false;
  }

  auto txn = exec_ctx_->GetTransaction();
  auto txn_mgr = exec_ctx_->GetTransactionManager();
  auto indexes = exec_ctx_->GetCatalog()->GetTableIndexes(table_info_->name_);
  auto table_heap = table_info_->table_.get();
  const auto &schema = table_info_->schema_;

  std::vector<std::pair<Tuple, RID>> rows_to_update;

  std::vector<Tuple> child_tuples;
  std::vector<RID> child_rids;
  while (child_executor_->Next(&child_tuples, &child_rids, batch_size)) {
    for (size_t i = 0; i < child_tuples.size(); i++) {
      rows_to_update.emplace_back(child_tuples[i], child_rids[i]);
    }
  }

  int32_t update_cnt = 0;

  for (const auto &[visible_old_tuple, rid] : rows_to_update) {
    Tuple target_tuple = MakeUpdatedTuple(
        schema,
        child_executor_->GetOutputSchema(),
        visible_old_tuple,
        plan_);

    if (IsTupleContentEqual(visible_old_tuple, target_tuple)) {
      continue;
    }

    auto [base_meta, base_tuple, undo_link] =
        GetTupleAndUndoLink(txn_mgr, table_heap, rid);

    if (IsWriteWriteConflict(base_meta, txn)) {
      TaintAndThrow(txn, "write-write conflict in update");
    }

    TupleMeta new_meta{txn->GetTransactionTempTs(), false};
    std::optional<UndoLink> new_undo_link = undo_link;

    if (base_meta.ts_ == txn->GetTransactionTempTs()) {
      // 当前事务已经写过这个 RID：更新已有 undo log。
      if (undo_link.has_value() && undo_link->IsValid() &&
          undo_link->prev_txn_ == txn->GetTransactionId()) {
        auto old_log = txn->GetUndoLog(undo_link->prev_log_idx_);
        auto updated_log = GenerateUpdatedUndoLog(
            &schema,
            base_meta.is_deleted_ ? nullptr : &base_tuple,
            &target_tuple,
            old_log);
        txn->ModifyUndoLog(undo_link->prev_log_idx_, updated_log);
      }
    } else {
      // 当前事务第一次写这个 RID：追加新 undo log。
      auto log = GenerateNewUndoLog(
          &schema,
          base_meta.is_deleted_ ? nullptr : &base_tuple,
          &target_tuple,
          base_meta.ts_,
          undo_link.value_or(UndoLink{}));
      new_undo_link = txn->AppendUndoLog(log);
    }

    bool ok = UpdateTupleAndUndoLink(
        txn_mgr,
        rid,
        new_undo_link,
        table_heap,
        txn,
        new_meta,
        target_tuple,
        [&](const TupleMeta &latest_meta, const Tuple &latest_tuple,
            RID latest_rid, std::optional<UndoLink> latest_link) {
          return !IsWriteWriteConflict(latest_meta, txn);
        });

    if (!ok) {
      TaintAndThrow(txn, "write-write conflict in update");
    }

    // 索引维护：先保守地删旧 key 插新 key。
    DeleteIndexEntries(indexes, schema, visible_old_tuple, rid, txn);
    InsertIndexEntries(indexes, schema, target_tuple, rid, txn);

    txn->AppendWriteSet(table_info_->oid_, rid);
    update_cnt++;
  }

  tuple_batch->emplace_back(
      std::vector<Value>{ValueFactory::GetIntegerValue(update_cnt)},
      &GetOutputSchema());

  has_updated_ = true;
  return true;
}
```

## 8. 一个更稳的实现顺序

建议按这个顺序做，不要一口气写完：

1. 在 `execution_common.h/.cpp` 里补 `IsTempTs`、`IsSelfModification`、`IsWriteWriteConflict`、`TaintAndThrow`。
2. 补 `MakeUpdatedTuple`。
3. 补 `DeleteIndexEntries` 和 `InsertIndexEntries`。
4. 改 `DeleteExecutor`，先不优化索引 key 是否变化。
5. 改 `UpdateExecutor`，先 materialize child 输出，再写。
6. 跑 `txn_executor_test` 里 undo log 相关测试。
7. 打开或局部运行 update/delete 相关 disabled 测试，配合 `TxnMgrDbg` 看版本链。
8. 如果索引测试失败，再回头检查 index scan 是否也按 MVCC 可见性读 tuple。

## 9. 常见坑

### 9.1 用 child tuple 生成 undo log

不要这样做。undo log 应该基于 table heap 当前 base tuple 生成，因为 undo link 链接的是版本链头部。

### 9.2 忘了二次检查冲突

你先 `GetTupleAndUndoLink`，再生成 undo log，中间可能有别的线程写了同一个 RID。`UpdateTupleAndUndoLink` 的 check lambda 就是用来防这个窗口的。

### 9.3 自己修改自己时 append 了多个 undo log

测试会检查：

```text
one transaction holds at most one undo log for one RID
```

同一个事务连续：

```sql
UPDATE table1 SET b = 2;
UPDATE table1 SET b = 3;
DELETE FROM table1;
```

应该还是一个 undo log，只是这个 undo log 记录的“原始旧版本”越来越完整。

### 9.4 Update 生成了新 RID

Project 4 的 update 不应该让 table heap entry 数量膨胀。测试里会有 `TableHeapEntryNoMoreThan`。

### 9.5 忘记写 write set

commit 现在靠 write set 找到当前事务改过哪些 RID，然后把临时时间戳改成 commit timestamp。忘记 `AppendWriteSet` 会导致提交后 tuple 仍然带着临时时间戳。

### 9.6 IndexScan 仍然按 Project 3 读

你当前 `IndexScanExecutor` 还是：

```cpp
auto [meta, tuple] = table_info_->table_->GetTuple(rid);
if (meta.is_deleted_) continue;
```

等做索引相关测试时，它也需要像 `SeqScanExecutor` 一样使用：

```cpp
CollectUndoLogs(rid, meta, tuple, txn_mgr->GetUndoLink(rid), txn, txn_mgr)
ReconstructTuple(&table_info_->schema_, tuple, meta, undo_logs)
```

否则普通 seq scan 可能对，index scan 却读错版本。

## 10. 最小验收标准

你可以用这几个现象判断方向对不对：

- `UPDATE` 后 RID 不变，table heap entry 数量不增加。
- 当前事务能看到自己的 update/delete。
- 老事务仍然能看到它 read_ts 对应的旧版本。
- 其他事务未提交写入会导致当前写事务 tainted。
- 当前事务重复修改同一个 RID，不会增加多个 undo log。
- commit 后，write set 中 RID 的 meta ts 从 temp ts 变成 commit ts。
- delete 不是物理删除，而是最新版本变成 delete marker，旧版本通过 undo log 可恢复。
